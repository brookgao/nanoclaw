# 完整数据管道规范

> 数据管道必须端到端完整，不能有断点；每个阶段的输入输出必须明确

## 问题

数据管道中某一环的输出格式变更，但下游环节未更新，导致静默失败：数据看似在流动，但在某个中间节点被悄悄丢弃或转换为错误格式。

## 设计原则

在实现前，先画出数据从 A 到 Z 的完整路径：

```
用户输入 → 前端处理 → API 接收 → 业务逻辑 → DB 写入
                                         ↓
                                    Redis 发布
                                         ↓
                               SSE 推送 → 前端渲染
```

每个阶段必须明确：输入类型和格式、输出类型和格式、异常处理策略。

## 常见断点

- DB 字段为 None 但下游不处理 None
- JSON 序列化后字段名拼写变化（camelCase vs snake_case）
- 列表为空但下游期望至少一个元素

## 教训

- 数据管道中任何一环断开，下游全部失效且通常静默失败
- 设计数据管道时，先画出完整链路图，再实现
- 每个阶段必须有明确的输入类型、输出类型和错误处理
